# План для нового rule: разметка legacy-проекта

## 1. Назначение нового rule

Новый rule должен описывать полный, повторяемый и проверяемый процесс приведения legacy-проекта к формату, ожидаемому текущими `.skills` rules:

- разметить код семантическими блоками и контрактами;
- создать обязательные XML-артефакты, которые прямо фигурируют в текущих rules;
- выполнить автоматическую проверку полноты и консистентности разметки;
- выполнить smoke test после внесения разметки.

## 2. Что именно должен покрывать rule

Rule должен работать не только с новым кодом, но и с уже существующим legacy-кодом, где:

- отсутствуют `MODULE_CONTRACT`, `MODULE_MAP`, `USE_CASES`;
- нет `START_FUNCTION_*`, `END_FUNCTION_*`, `START_CLASS_*`, `END_CLASS_*`;
- отсутствуют логические `START_BLOCK_*` / `END_BLOCK_*`;
- нет `AppGraph.xml`;
- нет резервной XML-копии knowledge graph;
- отсутствует тест, подтверждающий, что разметка не сломала проект и покрывает весь целевой scope.

## 3. Явные XML-артефакты из текущих rules

По текущему набору rules новый процесс должен уметь создавать или обновлять:

- `AppGraph.xml`
- `Knowledgebase.xml`

Уточнение:

- `KnowledgeGraph` в `.skills/rules-code/graph_generation_prompt.md` является корневым XML-тегом, а не именем отдельного файла.
- `DraftCodeGraph` из шаблона development plan является частью документа-плана, а не обязательным standalone XML-файлом проекта.

## 4. Цели нового rule

- Зафиксировать пошаговый процесс полной разметки legacy-проекта.
- Разделить работу на инвентаризацию, разметку, генерацию XML и проверку.
- Ввести quality gates, чтобы агент не объявлял задачу завершенной без проверки покрытия и валидности XML.
- Сделать процесс масштабируемым для больших проектов через пакетную обработку по модулям.

## 5. Предлагаемая структура нового rule

Новый rule логично разместить в:

- `.skills/rules-legacy/rules.md`

Дополнительно рядом имеет смысл завести:

- `.skills/rules-legacy/annotation_checklist.md`
- `.skills/rules-legacy/test_spec.md`
- `.skills/rules-legacy/xml_validation_guide.md`

## 6. Основной workflow, который должен описывать новый rule

### Этап 1. Scope и инвентаризация legacy-проекта

Rule должен заставлять агента сначала зафиксировать scope:

- определить корневые директории проекта;
- определить поддерживаемые языки;
- исключить `node_modules`, `.git`, build-output, generated files, vendor-код, миграции и артефакты окружения;
- построить реестр файлов, которые должны быть размечены;
- разделить файлы на категории:
  - entrypoints;
  - domain/core modules;
  - infrastructure;
  - helpers/utilities;
  - tests.

Выход этапа:

- список in-scope файлов;
- список исключений;
- приоритет обработки модулей.

### Этап 2. Анализ архитектуры legacy-проекта

До начала массовой разметки rule должен требовать:

- выделить точки входа;
- определить ключевые бизнес-сценарии;
- восстановить связи между модулями;
- зафиксировать внешние зависимости, базы данных, API, очереди, файлы;
- составить первичную карту вызовов и ответственности модулей.

Выход этапа:

- черновая архитектурная карта;
- список доменных модулей;
- список ключевых use case.

### Этап 3. Разметка исходного кода

Rule должен требовать пакетную разметку legacy-кода сверху вниз:

1. Для каждого модуля:
- добавить `MODULE_CONTRACT`;
- добавить `MODULE_MAP`;
- добавить `USE_CASES`, если модуль участвует в значимых сценариях;
- описать scope, inputs, outputs, keywords, links.

2. Для каждого публичного класса и функции:
- добавить `START_FUNCTION_*` / `END_FUNCTION_*` или `START_CLASS_*` / `END_CLASS_*`;
- добавить `CONTRACT`;
- синхронизировать contract с фактической сигнатурой и поведением.

3. Для сложной внутренней логики:
- разбить код на `START_BLOCK_*` / `END_BLOCK_*`;
- сохранять линейность и читаемость;
- не вводить декоративную разметку там, где блок не несет отдельного смысла.

4. Для логирования:
- сохранить существующие полезные логи;
- привести новые и измененные логи к единому формату там, где это оправдано;
- не превращать разметку в тотальную перепись работающего модуля без необходимости.

### Этап 4. Генерация XML-артефактов

Rule должен требовать обязательного создания и обновления XML:

1. `AppGraph.xml`
- строится по фактическому состоянию legacy-кода после разметки;
- включает проектный уровень, модули, классы, методы, связи и сценарии;
- использует соглашения из `.skills/rules-code/graph_generation_prompt.md`;
- должен покрывать весь in-scope код, а не только измененные вручную файлы.

2. `Knowledgebase.xml`
- создается как XML-резервная копия графа знаний;
- если `memory` MCP доступен, источник истины берется из него;
- если `memory` MCP недоступен, rule должен явно пометить шаг как blocked или fallback-to-artifacts, а не молча пропустить артефакт.

### Этап 5. Валидация разметки

Rule должен запрещать завершение задачи без автоматической валидации:

- проверка, что все in-scope файлы либо размечены, либо явно исключены;
- проверка, что `MODULE_MAP` и фактические сущности не расходятся;
- проверка парности всех `START_*` / `END_*` тегов;
- проверка, что XML well-formed;
- проверка уникальности XML-тегов и корректности cross-links;
- проверка, что пути в `FILE="..."` реально существуют;
- проверка, что ссылки на методы/классы указывают на реальные узлы.

### Этап 6. Тест

Rule должен завершаться обязательным тестовым прогоном в двух слоях.

1. Структурный test:
- validation script проверяет разметку и XML;
- выдает coverage report по файлам и сущностям.

2. Smoke test:
- запускается минимальный сценарий проекта;
- подтверждает, что разметка и служебные комментарии не сломали импорт, сборку или запуск;
- если проект без формализованных тестов, smoke test становится обязательным минимальным качественным барьером.

## 7. Какой test нужен для этого rule

Минимальный тестовый набор для legacy-markup rule:

### Test A. Annotation Coverage Test

Проверяет:

- каждый in-scope файл присутствует в inventory;
- каждый модуль имеет обязательный верхнеуровневый блок разметки;
- публичные функции и классы имеют semantic anchors;
- непокрытые файлы попадают в явный отчет.

### Test B. XML Integrity Test

Проверяет:

- существует `AppGraph.xml`;
- существует `Knowledgebase.xml` или корректно зафиксирован статус `blocked`;
- XML парсится без ошибок;
- все `TARGET` ссылки резолвятся;
- нет дубликатов XML-идентификаторов;
- нет ссылок на несуществующие файлы.

### Test C. Runtime Smoke Test

Проверяет:

- проект импортируется;
- базовый entrypoint стартует;
- критичные зависимости не сломаны комментариями и рефакторингом разметки;
- логирование не вызывает runtime error.

## 8. Пошаговый план внедрения нового rule

### Шаг 1. Создать сам rule

Создать:

- `.skills/rules-legacy/rules.md`

Внутри описать:

- purpose;
- inputs;
- outputs;
- пошаговый workflow;
- quality gates;
- stop conditions;
- требования к XML;
- требования к test.

### Шаг 2. Добавить checklist

Создать:

- `.skills/rules-legacy/annotation_checklist.md`

Он должен использоваться как операционный контрольный список:

- inventory готов;
- exclusions утверждены;
- разметка по модулям завершена;
- `AppGraph.xml` создан;
- `Knowledgebase.xml` создан или помечен blocked;
- test green.

### Шаг 3. Добавить test-spec

Создать:

- `.skills/rules-legacy/test_spec.md`

В нем зафиксировать:

- входные данные для validation;
- expected outputs;
- критерии fail/pass;
- формат итогового отчета.

### Шаг 4. Добавить validator script

Реализовать отдельный validator, который проверяет:

- полноту разметки;
- парность semantic tags;
- целостность XML;
- покрытие inventory.

Предпочтительный результат:

- один запускаемый script, который может быть использован и в CI, и локально.

### Шаг 5. Прогнать pilot на одном модуле

До полного rollout новый rule должен быть проверен на одном representative legacy-модуле:

- средний по размеру файл;
- есть функции, классы и внешние вызовы;
- можно построить use cases и блоки.

### Шаг 6. Масштабировать на весь проект

После успешного pilot:

- размечать пакетами по директориям;
- после каждого пакета обновлять `AppGraph.xml`;
- в конце формировать финальный `Knowledgebase.xml`;
- прогонять structural validation и smoke test.

## 9. Критерии готовности

Новый rule можно считать готовым, если:

- описан полный workflow для legacy-разметки;
- rule покрывает инвентаризацию, разметку, XML и тест;
- явно перечислены обязательные XML-артефакты;
- есть validator strategy;
- есть smoke-test strategy;
- есть понятные conditions for done;
- rule применим к большому проекту поэтапно, а не только к маленькому demo-кейсу.

## 10. Практические ограничения, которые rule должен проговаривать явно

- Нельзя размечать third-party и generated code.
- Нельзя считать задачу завершенной только по факту вставки комментариев.
- Нельзя создавать `AppGraph.xml` по догадкам без сверки с реальными файлами.
- Нельзя молча пропускать `Knowledgebase.xml`, если он обязателен для выбранного сценария.
- Нельзя делать глобальную разметку без inventory и explicit exclusions.
- Нельзя пропускать test после массовой разметки legacy-кода.

## 11. Рекомендуемый результат следующего шага

После утверждения этого плана следующий практический шаг:

1. создать `.skills/rules-legacy/rules.md`;
2. сразу зафиксировать там workflow из разделов 6-10;
3. затем добавить `annotation_checklist.md`;
4. затем описать и реализовать validator/test.

# PRD: CITYRNNG

## 1. Контекст

`CITYRNNG` - городской беговой проект. Цель платформы - дать участникам единое цифровое пространство, где можно:

- создать аккаунт;
- записываться на пробежки;
- подтверждать участие;
- копить баллы;
- обменивать баллы на предложения партнеров;
- видеть свой прогресс и историю активности.

Платформа должна усиливать сообщество, а не превращать проект в сухой спортивный сервис.

## 2. Цели продукта

### Бизнес-цели

- собрать собственную базу пользователей вне Instagram;
- повысить возвратность на регулярные пробежки;
- создать прозрачную систему партнерских интеграций;
- дать команде проекта данные по участникам, событиям и rewards;
- подготовить фундамент для масштабирования на другие города.

### Пользовательские цели

- быстро понять, как присоединиться;
- без трения зарегистрироваться;
- видеть ближайшие пробежки;
- получать баллы за участие;
- понимать, на что можно обменять баллы;
- чувствовать принадлежность к городскому беговому комьюнити.

## 3. North Star и KPI

### North Star Metric

- количество подтвержденных участий в пробежках в месяц

### Product KPI

- регистрации в платформе;
- MAU;
- доля пользователей, пришедших минимум на 2 пробежки за 30 дней;
- среднее число пробежек на активного пользователя;
- redemption rate rewards;
- количество активных партнеров;
- конверсия `visit -> signup -> first event check-in`;
- retention D30 по пользователям, завершившим первую пробежку.

## 4. Целевая аудитория

### Основная

- городские жители 20-35 лет;
- новички и любители бега;
- люди, которым важны атмосфера, знакомства, стиль города и регулярный ритуал.

### Вторичная

- партнеры: кофейни, рестораны, wellness, локальные бренды, event-площадки;
- event-менеджеры и организаторы забегов;
- амбассадоры сообщества.

## 5. Роли

- `Guest` - незарегистрированный пользователь
- `Runner` - обычный участник сообщества
- `Partner Manager` - представитель партнера
- `Event Manager` - организатор пробежек / событий
- `Admin` - операционное управление платформой
- `Superadmin` - полный доступ, системные настройки

## 6. MVP Scope

### Входит в MVP

- публичный сайт;
- регистрация и вход;
- личный кабинет участника;
- каталог событий;
- подтверждение участия через Strava-синк активности (QR-поток отменён в пользу этого подхода);
- начисление баллов;
- каталог rewards / предложений партнеров;
- списание баллов;
- админка для пользователей, событий, rewards, партнеров и ручных корректировок;
- базовые уведомления по email;
- базовая аналитика и audit log.

### Не входит в MVP

- мобильное приложение;
- интеграция с Garmin / Apple Health (Strava — в MVP, см. выше);
- сложный social graph;
- встроенный чат;
- динамическое ценообразование;
- полноценный marketplace с оплатами;
- multi-city и мультибрендовый режим;
- сложная антифрод-система на ML.

## 7. Продуктовые принципы

- Простая первая ценность: пользователь должен понять пользу за 30 секунд.
- Минимум ручных действий на пробежке.
- Баллы должны ощущаться достижимыми.
- Интерфейс должен передавать городской lifestyle, а не медицинский фитнес.
- Вся критичная логика баллов должна быть аудируема.

## 8. Основные пользовательские сценарии

### Сценарий 1. Первый вход

1. Пользователь заходит на сайт.
2. Видит ближайшую пробежку, преимущества, механику баллов и rewards.
3. Регистрируется по email.
4. Получает welcome-баллы.
5. Попадает в личный кабинет.

### Сценарий 2. Запись на пробежку

1. Пользователь открывает список событий.
2. Выбирает ближайшую пробежку.
3. Видит дату, время, локацию, темповые группы, дистанции, ограничения.
4. Нажимает "Записаться".
5. Событие появляется в профиле.

### Сценарий 3. Подтверждение участия и начисление баллов

1. Пользователь однократно подключает Strava в профиле (OAuth).
2. После пробежки админ запускает синк активностей, система импортирует их в `external_activities`.
3. Matcher сверяет активность с правилом события (окно, distance/duration, геофенс через `city_locations`) и создаёт `event_attendance`.
4. Если правило `auto_approve=true` — attendance сразу approved, иначе админ подтверждает вручную.
5. При переходе в approved идемпотентно начисляются баллы, пользователь видит транзакцию в истории.

### Сценарий 4. Обмен баллов

1. Пользователь открывает каталог rewards.
2. Видит стоимость в баллах, ограничения и срок действия.
3. Нажимает "Обменять".
4. Система резервирует или списывает баллы.
5. Создается redemption с кодом / QR / подтверждением для партнера.

### Сценарий 5. Работа администратора

1. Администратор создает событие.
2. Запускает check-in.
3. Смотрит участников и проблемные записи.
4. Начисляет ручной бонус при необходимости.
5. Управляет партнерами и rewards.

## 9. Балльная механика MVP

Рекомендуемые правила на старт:

- регистрация: `+50`
- первая подтвержденная пробежка: `+150`
- каждая обычная подтвержденная пробежка: `+100`
- участие в спецсобытии: `+150..300`
- друг, дошедший до первой пробежки по приглашению: `+200`
- ручной бонус от админа: настраиваемый

Принципы:

- операции только через журнал транзакций;
- никакого прямого редактирования общего баланса;
- повторное начисление по одному и тому же attendance запрещено (idempotency key `event_attendance:<attendanceId>`);
- отмены должны идти обратной транзакцией.

## 11. Контентные требования

Тон бренда:

- городской;
- живой;
- дружелюбный;
- современный;
- без спортивного снобизма.

На главной обязательно должны быть:

- ближайшее событие;
- объяснение механики "бегаешь -> получаешь баллы -> обмениваешь";
- партнеры;
- фото / атмосфера сообщества;
- CTA на регистрацию и запись на ближайшую пробежку.

## 12. Ограничения и риски

- нельзя строить продукт как набор разрозненных landing pages;
- нельзя хранить баллы в виде одного изменяемого поля без истории;
- нельзя завязывать бизнес-процесс только на ручные действия админа;
- не стоит сразу строить мобильное приложение до подтверждения retention;
- подтверждение участия должно быть защищено от повторов и накрутки (уникальность `event_attendance` по `(event_id, user_id)` + idempotency начисления баллов).

## 13. Критерии успеха MVP

- пользователь может зарегистрироваться и подключить Strava без помощи менеджера;
- event manager может подтвердить pending-attendance в админке;
- после пробежки баллы начисляются корректно (автоматически при auto-approve или после ручного approve);
- reward можно обменять без ручного вмешательства разработчика;
- администратор может проследить всю историю начислений и списаний;
- команда получает минимальную аналитику по активности и rewards.
